# 개인 소비 데이터 분석 시스템  
## 기능 요구서 v3 (분석 시나리오 중심 문서)

---

## 1. 문서 목적

본 문서는 개인 소비 데이터 분석 시스템의 개발 방향을 정의하는 **분석 시나리오 중심 요구사항 문서**이다.

이 시스템은 단순 조회 도구가 아니라,  
전처리된 소비 데이터를 바탕으로 **소비 패턴을 해석하고, 변화 원인을 찾고, 특정 조건의 결과를 비교하고, 의미 있는 분석 결과를 도출하는 분석 엔진**을 목표로 한다.

따라서 본 문서는 다음을 중요하게 본다.

- 기능 개수보다 **분석 가능성**
- 단일 함수보다 **재사용 가능한 계산 구조**
- 결과 출력보다 **해석 가능한 분석 흐름**
- 구현 편의보다 **확장 가능한 설계 방향**

---

## 2. 프로젝트 핵심 목표

본 시스템은 사용자가 자신의 소비 데이터를 보고 다음과 같은 질문에 답할 수 있도록 설계되어야 한다.

- 이번 달에 어디에 가장 많이 썼는가?
- 지난달보다 무엇이 얼마나 늘었는가?
- 특정 카테고리의 소비가 왜 커졌는가?
- 지출 구조에서 어떤 카테고리가 가장 큰 비중을 차지하는가?
- 내가 평소보다 많이 쓴 항목은 무엇인가?
- 특정 조건의 데이터만 따로 묶어서 보면 어떤 패턴이 보이는가?

즉, 이 시스템의 목표는  
**데이터를 보여주는 것**이 아니라  
**데이터를 해석 가능한 형태로 바꾸는 것**이다.

---

## 3. 시스템 구성 방향

본 시스템은 크게 두 영역으로 나뉜다.

### 3-1. 데이터 전처리 영역
원본 CSV를 분석 가능한 상태로 정리한다.

### 3-2. 데이터 분석 영역
정리된 데이터를 대상으로 조회, 요약, 비교, 통계, 시각화 기능을 재조합하여  
사용자의 소비 패턴을 분석한다.

---

## 4. 전처리 요구사항

전처리 영역은 기존과 동일하게 안정성을 우선한다.

### 필수 요구사항

- CSV 파일 로드 가능
- UTF-8-SIG 인코딩 대응
- 날짜 형식 정리
- 타입 정리 (`income`, `refund` → `수입`, `expense` → `지출`)
- 카테고리 정리
- 금액 숫자형 변환
- 문제 행 분리
- 분석용 데이터셋 반환

### 전처리 결과가 만족해야 할 조건

- `date`는 datetime 타입이어야 한다
- `year`, `month`, 필요 시 `year_month` 컬럼이 생성되어야 한다
- `type`은 `수입`, `지출`만 남아야 한다
- `amount`는 숫자형이어야 한다
- 분석 제외 대상 행은 따로 확인할 수 있어야 한다

---

## 5. 분석 모델 설계 원칙

### 5-1. 분석 기능은 조합형 구조여야 한다
분석 결과는 하나의 거대한 메서드로 직접 만들어내는 것이 아니라,  
다음과 같은 작은 단위 기능들을 조합해서 만들어야 한다.

- 조건 조회
- 집계/요약
- 비교
- 통계 계산
- 화면용 정렬/출력 가공

### 5-2. 조회 결과는 다른 분석의 입력이 될 수 있어야 한다
예를 들어 다음 흐름이 자연스럽게 가능해야 한다.

- 특정 월 조회 → 카테고리별 요약
- 특정 타입 조회 → 상위 지출 분석
- 특정 카테고리 조회 → 월 간 비교
- 특정 기간 조회 → 평균/비중 계산

### 5-3. 반환 구조는 일관되어야 한다
대부분의 분석 결과는 `DataFrame` 또는 `Series` 형태로 유지하여  
다른 계산이나 시각화에 다시 사용할 수 있어야 한다.

### 5-4. 모델은 “완성된 화면”보다 “분석 재료”를 우선 제공해야 한다
화면에서 보기 좋은 형태는 필요하지만,  
그보다 먼저 **다른 메서드에서 다시 쓸 수 있는 중간 결과**를 안정적으로 제공해야 한다.

---

## 6. 핵심 분석 시나리오

아래 항목은 단순 기능 목록이 아니라,  
시스템이 실제로 해결할 수 있어야 하는 분석 질문들이다.

---

## 6-1. 특정 월 소비 구조 분석

### 사용자 질문
- 이번 달에 어디에 가장 많이 썼는가?
- 이번 달 수입/지출 구조는 어떤가?
- 카테고리별로 얼마나 썼는가?

### 시스템이 제공해야 하는 분석
- 특정 연/월 데이터 조회
- 해당 월의 총수입 / 총지출 / 순이익 요약
- 해당 월의 카테고리별 지출 합계
- 해당 월의 카테고리별 건수
- 해당 월의 카테고리별 비중
- 필요 시 상위 지출 내역 조회

### 설계 의도
이 시나리오는 단일 메서드 하나로 끝나는 것이 아니라,  
조회 → 요약 → 비중 계산 → 상위 내역 추출 흐름으로 조합 가능해야 한다.

---

## 6-2. 월 간 변화 분석

### 사용자 질문
- 지난달보다 이번 달에 무엇이 얼마나 늘었는가?
- 수입/지출/순이익이 어떻게 변했는가?
- 어떤 카테고리가 가장 크게 증가했는가?

### 시스템이 제공해야 하는 분석
- 두 달의 요약 비교
- 두 달의 카테고리별 합계 비교
- 증감 계산
- 증감률 계산
- 증가폭 상위 카테고리 정렬
- 감소폭 상위 카테고리 정렬

### 설계 의도
비교 기능은 “기준월과 비교월을 나열하는 것”에서 끝나지 않고,  
**변화량을 해석하는 구조**까지 포함해야 한다.

---

## 6-3. 특정 카테고리 추적 분석

### 사용자 질문
- 식비는 몇 달 동안 어떻게 변해왔는가?
- 카페 지출은 최근에 늘고 있는가?
- 특정 카테고리는 어느 달에 가장 컸는가?

### 시스템이 제공해야 하는 분석
- 특정 카테고리 데이터만 추출
- 월별 흐름 요약
- 월 간 비교
- 평균 대비 증감 확인
- 상위 지출 내역 확인

### 설계 의도
카테고리 분석은 단순 필터링 기능이 아니라,  
**특정 주제를 잡고 흐름을 추적하는 분석 단위**로 사용되어야 한다.

---

## 6-4. 지출 비중 분석

### 사용자 질문
- 내 지출에서 가장 큰 비중을 차지하는 항목은 무엇인가?
- 특정 달의 소비 구조는 어느 쪽에 치우쳐 있는가?

### 시스템이 제공해야 하는 분석
- 전체 지출 대비 카테고리별 비율 계산
- 특정 월 지출 대비 카테고리별 비율 계산
- 지출 비중 상위 카테고리 정렬
- 필요 시 원그래프/막대그래프에 바로 연결 가능한 결과 반환

### 설계 의도
비중 분석은 단순 합계가 아니라,  
**전체 안에서 각 항목이 차지하는 상대적 위치**를 보여줘야 한다.

---

## 6-5. 평소보다 큰 소비 탐지

### 사용자 질문
- 내가 평소보다 크게 쓴 항목은 무엇인가?
- 평균보다 많이 쓴 카테고리는 어디인가?
- 유독 큰 금액이 발생한 내역은 무엇인가?

### 시스템이 제공해야 하는 분석
- 카테고리별 평균 금액 계산
- 특정 기간 또는 전체 기준 평균과 실제 값 비교
- 상위 금액 추출
- 평균 대비 초과 금액 또는 배수 계산

### 설계 의도
평균 기능은 단독 기능으로 쓰기보다,  
**이상치 탐지 또는 과소비 탐지의 기준값**으로 활용될 때 의미가 커진다.

---

## 6-6. 조건 조합 분석

### 사용자 질문
- 이번 달 식비 중 2만 원 이상만 보고 싶다
- 지출 중 특정 키워드가 들어간 내역만 보고 싶다
- 특정 기간의 카페 지출만 따로 요약하고 싶다

### 시스템이 제공해야 하는 분석
- 연/월 조건 조회
- 기간 조건 조회
- 타입 조건 조회
- 카테고리 조건 조회
- 최소 금액 조건 조회
- 키워드 조건 조회
- 조회 결과를 다시 요약/비교/통계 계산에 전달 가능해야 함

### 설계 의도
조회 기능은 독립 기능이 아니라,  
**복합 분석을 만들기 위한 입력 데이터 생성기**여야 한다.

## 8. 우선순위 재정의

이 문서에서는 기능 우선순위를 다음과 같이 정의한다.

### 1순위: 전처리 안정성
잘못된 데이터가 분석으로 섞여 들어가지 않아야 한다.

### 2순위: 조합 가능한 조회/요약 구조
조건 조회 결과를 다른 기능에서 바로 재사용할 수 있어야 한다.

### 3순위: 변화량 해석 기능
단순 수치보다 증감, 증감률, 비중 등 해석 가능한 값이 중요하다.

### 4순위: 통계 기능의 실사용성
평균은 단독 기능으로 끝내지 말고, 비교 기준값으로 재사용할 수 있어야 한다.

### 5순위: 시각화 연결성
그래프는 마지막 단계지만, 결과 데이터는 그래프에 바로 연결 가능해야 한다.

---

## 9. 시각화 요구사항 (선택)

시각화는 필수는 아니지만, 분석 모델의 결과를 표현하는 단계로 확장 가능해야 한다.

### 권장 시각화
- 월별 수입/지출 막대그래프
- 월별 순이익 선그래프
- 특정 월 카테고리 비중 원그래프
- 상위 지출 막대그래프
- 특정 카테고리 월별 추이 그래프

### 요구사항
- 그래프 함수는 직접 그릴 수도 있고
- 그래프용 데이터만 반환해도 무방하다
- 중요한 것은 **분석 모델 결과가 시각화와 잘 연결되는 구조**를 가지는 것이다

## 11. 검수 기준

개발 완료 후 아래 항목을 확인할 수 있어야 한다.

### 전처리 검수
- CSV가 정상적으로 로드되는가
- 날짜가 datetime으로 정리되는가
- 타입이 `수입`, `지출`로만 정리되는가
- 금액이 숫자형으로 정리되는가
- 문제 행을 따로 확인할 수 있는가

### 분석 검수
- 조회 결과를 다른 메서드에 다시 사용할 수 있는가
- 월별/카테고리별 요약이 정확한가
- 비교 결과에 증감과 증감률이 포함되는가
- 특정 카테고리 추적 분석이 가능한가
- 비중 계산 결과가 정확한가
- 평균값이 실제 분석 시나리오에서 활용 가능한가

### 구조 검수
- 메서드 간 재사용성이 있는가
- 기능 추가 시 기존 구조를 크게 뜯지 않아도 되는가
- 화면 출력용 가공과 내부 계산 로직이 분리되어 있는가

---

## 12. 최종 기대 결과

최종 결과물은 다음을 만족해야 한다.

1. 원본 CSV를 안정적으로 전처리할 수 있다  
2. 분석용 데이터셋과 문제 데이터셋을 분리할 수 있다  
3. 조회 / 요약 / 비교 / 통계 기능을 조합하여 다양한 분석 결과를 만들 수 있다  
4. 사용자가 소비 패턴을 해석할 수 있는 결과를 제공할 수 있다  
5. 평균, 비중, 증감 등의 값이 실제 분석 맥락에서 활용될 수 있다  
6. 이후 시각화, UI, API 구조로 확장 가능한 형태를 가진다

## 13. 설계 철학 요약

이 시스템은 단순한 기능 모음이 아니다.

이 시스템은  
**전처리된 소비 데이터를 바탕으로 분석 질문에 답하기 위한 조합형 분석 엔진**이어야 한다.

따라서 좋은 구현이란  
함수를 많이 만드는 것이 아니라,

- 재사용 가능한 계산 단위를 만들고
- 그 단위들을 조합하여
- 분석 질문에 답할 수 있게 만드는 것이다.
